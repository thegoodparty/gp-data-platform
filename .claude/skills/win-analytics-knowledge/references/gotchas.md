# Known gotchas reference

Part of the **win-analytics-knowledge** skill. Recurring traps, as a quick-reference symptom table.

## Quick reference

- **What this is:** a symptom → mitigation index. When a trap is fully explained by its home doc (joins, outcomes, viability, etc.), this table carries the **symptom** and points there rather than restating the fact — so a fact lives in exactly one place.
- **How to use it:** scan the Symptom column for what you're seeing, then follow the mitigation's link for the full explanation.
- **Maintenance:** when you hit a new trap, add a one-liner here; put the full explanation in the owning domain doc.

## Reading the Status column — these are not all permanent

Each entry is tagged:

- **`invariant`** — a methodological or structural truth that does not drift (a statistical fact, a data-model property, a policy). Trust it.
- **`state · as-of YYYY-MM`** — a data- or code-state fact with a shelf life. Treat it as a **hypothesis to re-verify against the live catalog**, not a standing truth — the older the as-of date, the more suspect. Some are designed to become false (e.g. a "fix ticketed" trap flips once the fix ships). The as-of date is the last point the fact was confirmed; the owning domain doc carries the fuller dated context.

The as-of dates are at month granularity on purpose (no false precision). A future linter (Phase 2 / Track A) will re-verify the mechanically checkable state entries automatically — catalog/column existence, schema types, the viability code/data discrepancy, the NULL `election_level` share, the onboarding-flag read — and flag the rest for a manual sweep once their as-of date is stale.

## Gotchas

| Gotcha | Symptom | Mitigation | Status |
|---|---|---|---|
| **`users_win_base.election_date = coalesce(next, last)`** | Leaks future election dates into "current cycle" analyses. | Use per-stage dates from `users_win_candidacy` instead. They are NOT coalesced and preserve cycle separation. | state · 2026-05 |
| **Corrupt `election_date` values (out-of-range years)** | A handful of latest-version non-demo rows (17 as of 2026-06) carry out-of-range `election_date` years like `+275760`, `+20021`, `0024`; the far-future ones pollute `MAX(election_date)` so those users read as perpetually upcoming — inflating any "election on/after date D" base — and they crash the non-arrow connector's date parser on return. | When aggregating election dates, bound to a sane window: `MAX(CASE WHEN election_date BETWEEN DATE'2020-01-01' AND DATE'2050-01-01' THEN election_date END)`. Return dates as STRING if corrupt rows can reach the client. Base recipe in [segmentation.md](segmentation.md). | state · 2026-06 |
| **`candidacy_result` cross-stage fallback** | A primary winner who lost the general can show `candidacy_result = 'Won'`. | Use `latest_stage_result` + `latest_stage_reached` (or `general_election_result`). See [outcomes.md](outcomes.md). | state · 2026-05 |
| **Viability score code/data discrepancy** | `candidacy.viability_score` is populated in prod but the SQL files hardcode it to NULL; future rebuilds could drop ~58k scores. | Join `int__techspeed_viability_scoring` via `techspeed_candidate_code` for forward-stable access. See [viability.md](viability.md). | state · 2026-05 |
| **Pre-2025-05-28 candidates have no Win-product Amplitude history** | Earlier `users.created_at` rows exist in the product DB but have no Win-Amplitude events (instrumentation went live ~2025-05-28; see [engagement.md](engagement.md)). | For engagement-window floors, use the **actual coverage `MIN(week_start_date)`** of the source table (currently 2025-06-23 for the weekly model), not a fixed date. Document the coverage window in the brief's `data_provenance` field. | state · 2026-05 |
| **45.6% NULL `election_level` in window** | Office-level slicing loses half the population if you filter. | Decide explicitly: (a) restrict, (b) backfill from `office_type` / `int__icp_offices`, or (c) treat NULL as a fifth category. See [segmentation.md](segmentation.md). | state · 2026-05 |
| **Pro users have LOWER raw win rate than free** | Naive Pro-lift analysis flips intuition. | Confounded by selection bias (Pro users self-select into harder races) and reverse causation (engagement → Pro). Office-stratify; use pre/post-upgrade frame for timing. | state · 2026-05 |
| **ICP=true cohort has LOWER raw win rate than ICP=false** | Same direction as the Pro finding. | ICP=true offices are by-design competitive. Slice rather than filter. See [segmentation.md](segmentation.md). | state · 2026-05 |
| **2025 archive vs 2026+ FOJ structural mismatch** | Field availability differs (e.g., `is_open_seat` absent from 2025 archive). | Slice by `source_systems` and `year(general_election_date)` together. Don't pool blindly. See [sources.md](sources.md). | state · 2026-05 |
| **Outcome data concentrated in 2025** | 2026 cycle has <10% outcome coverage. | "Did they win" analyses are effectively a 2025 cohort study for now. 2026 holdout is limited. | state · 2026-05 |
| **`votes_received` is STRING** | Casting fails on rows containing "uncontested". | Filter or coalesce the literal before casting. See [outcomes.md](outcomes.md). | state · 2026-05 |
| **Multi-cycle candidates** | One user can produce multiple `campaign_version_id` rows. | Filter to `is_latest_version` for the canonical row; collapse cycles explicitly. See [joins.md](joins.md). | invariant |
| **Amplitude self-reported outcome response bias** | 80% self-report "won" vs 64% raw population win rate. | Use labels, NOT rates. See [outcomes.md](outcomes.md). | state · 2026-05 |
| **L2 PII boundary** | Voter-grain L2 models carry PII-adjacent records. | Aggregate-grain queries only by default. See [sources.md](sources.md). | invariant |
| **Mart staleness vs SQL state** | Prod mart may carry data the current SQL doesn't produce (e.g., viability). | When code/data disagree, trust the data for current state but flag the discrepancy. Don't extrapolate forward. | invariant |
| **VIRTUAL_ENV breaks pre-commit pytest hook** | `Executable pytest not found` on commit when an analytics venv is active. | Deactivate the analytics venv (or run `git commit` from a clean shell) — the hook resolves pytest via the active `VIRTUAL_ENV`. | state · 2026-06 |
| **`users_win_candidacy.hubspot_id` is the COMPANY id, not the contact id** | Joining `stg_airbyte_source__hubspot_api_feedback_submissions.hs_contact_id` directly to `users_win_candidacy.hubspot_id` returns zero matches. | Use the 2-hop via candidate. See [joins.md](joins.md). | state · 2026-05 |
| **HubSpot PMF Option 1/2 labels obscured in export** | `pmf_response` returns literal `"Option 1"` / `"Option 2"` instead of the disappointment labels. | Treat **Option 1 = Very disappointed, Option 2 = Somewhat disappointed**. See [outcomes.md](outcomes.md). | state · 2026-05 |
| **Onboarding milestone flags are new-flow-blind** | `onboarding_complete` / `has_completed_onboarding_flow` / `is_onboarded` / the lib `onboarded` flag read FALSE for post-cutover users, so a cross-cutover analysis shows a fake completion collapse. | Recompute **onboarding dashboard viewed** (broad) or the era-resolved pledge union (strict; 3 events — `Onboarding - Candidate Pledge Completed` alone is V2-era-blind, dead after 2026-06-10). Mechanism, dates, and verified wrong-reads are in [engagement.md](engagement.md). | state · 2026-07 (fix ticketed — re-check) |
| **Account-creation column name** | `users_win_base.user_created_at` does NOT exist. | Account creation = `users_win_candidacy.user_created_at` (= `users_win_base.registered_at`, byte-identical). | state · 2026-05 |
| **Civics classification lag on recent signups** | Recent registrants carry NULL `election_level` / `office_type = 'Other'` until the civics mart classifies them, so office/level cuts on a recent cohort collapse. | Treat office/level stratification as a structural finding on the older classified cohort; don't read a recent pre/post office cut. | state · 2026-05 |
| **Amplitude coverage can lead the calendar date** | `MAX(event_time)` may be ahead of "today" (e.g. 2026-06-06 seen on 2026-06-02). | Compute `data_max = MAX(event_time)` live and derive censoring cutoffs from it; never hardcode "today" or a stale max. | invariant |
| **Closed-cohort retention declines by construction** | An election-anchored or signup-anchored closed cohort (membership fixed by first-active week) decays toward the anchor the way any product's weekly retention decays, and gets misread as the base disengaging as the election nears. On Win, the same dashboard-view metric falls in the closed-cohort curve but rises on an open view (8.6% to 15.2% for Nov-2025 generals). | When a retention curve could be read as aggregate engagement, pair it with an **open fixed-denominator** view: numerator = users active that week, denominator = the full population held constant. The closed curve answers whether early adopters stay sticky; the open curve answers whether the base is more engaged near the election. For the Win run-up question specifically, the recommended approach is to lead with the open view and then split it into **new** (first-ever active that week) versus **returning** (active this week and active in some earlier week), both on the fixed denominator. The returning line is the cleanest reverse-retention read: it separates real re-engagement from new signups, since `new% + returning%` equals the open active share. On Nov-2025 generals the returning share rose 3.6% to 10.8% while new stayed roughly flat near 5%, so the run-up was repeat engagement, not acquisition. Reference build: `analytics/ad_hoc/2026-06-02_win_retention_curve.ipynb`. | invariant (principle); embedded figures are illustrative · 2026-06 |
| **Moving or eligibility-filtered denominator makes a rate trend a composition artifact** | A cross-period rate's level AND trend move even if no individual user's behavior changed, because the denominator's composition is shifting. Mechanisms seen on Win: a cumulative-stock denominator against a signup-windowed numerator (old registrants' window closed, possibly pre-instrumentation, depressing the level; recent registrants right-censored, and the censored share varies by period, biasing the trend); a declining pre-instrumentation structural-zero share mechanically lifting the rate; upcoming-election eligibility churn. | **Hold the cohort/denominator fixed and re-measure; if the trend flattens, it was composition, not behavior.** To equalize windows: restrict the denominator to users whose full N-day window closed by the cut (signup ≤ cut − N days), or report the censored/ineligible N separately. The dual of the closed-cohort row above: that one is fixed membership decaying; this one is moving membership shifting. | invariant |
